# N2（query rewrite / multi-query）是要做什麼？可以改善什麼？

這份是你在 N2 前的提問整理版，聚焦 3 個問題：

1. N2 要做什麼？
2. 可以改善什麼？
3. rewrite / multi-query 會不會很耗 LLM 次數？

---

## 1) N2 要做什麼？

N2 的核心是：**提升檢索前的查詢品質**，讓後面的檢索更容易命中正確文件。

常見有兩條路：

- `query rewrite`：把原始問題改寫成更清楚、可檢索的查詢（通常 1 條）
- `multi-query`：從同一問題生成多個角度的查詢（例如 2~4 條），各自檢索後再合併

---

## 2) N2 可以改善什麼？

主要改善：

- **召回率（recall）**：降低「明明有資料但沒抓到」的情況
- **穩定性**：使用者問法不同時，結果不容易大幅波動
- **覆蓋度**：多面向問題更容易抓到不同子主題的證據

常見適用情境：

- 問題太口語、太短、太模糊
- 同義詞很多（術語說法不一致）
- 一句問題其實包含兩個以上子問題

---

## 3) rewrite 和 multi-query 都要用到 LLM 嗎？要很多次嗎？

### query rewrite

- 常見：**1 次 LLM** 產生 1 條改寫 query
- 也可不用 LLM（規則式），但通常效果較弱

### multi-query

- 常見：**1 次 LLM** 產生多條 query（例如 2~4 條）
- 成本常在後續的「多次檢索」，不一定是多次 LLM

結論：

- 不一定都要很多次 LLM
- 但 multi-query 通常會增加整體成本與延遲（因為要多跑 retrieval）

---

## 4) 你目前專案的建議（省 token 版本）

考量你有 429 / token 限制，建議順序：

1. 先做 `rewrite only`（每題多 1 次 LLM，成本最低）
2. 有提升再開 `multi-query=2`（先小步）
3. 保留開關與上限（像你現在控 finalize 次數）

---

## 5) 一句話記憶

N2 不是直接回答問題，而是先把「問法」變得更好，讓後面的檢索更容易找對資料。
