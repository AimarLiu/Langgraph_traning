# RRF（Reciprocal Rank Fusion）是什麼？怎麼做？需要模型嗎？

`RRF` 常出現在 **hybrid retrieval**（例如 BM25 + vector）要把兩張「排序表」合成一張最終排序時。

---

## 1) RRF 算不算啟發式？

算。**RRF 是一種啟發式融合排序（fusion）方法**：它用固定公式把多個檢索器各自的「排名」合成為一個新分數，再排序。

它通常 **不是**「從資料學出來的最優排序器」（那更接近 learning-to-rank / reranker），但在工程上很常見：**簡單、便宜、對分數量綱不敏感**。

---

## 2) RRF 的做法（最常見版本）

假設你有 \(m\) 個檢索器（或 \(m\) 張排序表），每一張表都各自排出候選文件的順序（rank）。

對每個文件 \(d\)，RRF 分數常寫成：

\[
\text{RRF}(d)=\sum_{i=1}^{m}\frac{1}{k+\text{rank}_i(d)}
\]

- \(i\)：第 \(i\) 個檢索器（例如 keyword / vector）
- \(\text{rank}_i(d)\)：文件 \(d\) 在第 \(i\) 張表的名次（通常從 **1** 開始）
  - 若 \(d\) 沒出現在第 \(i\) 張表：這一項通常視為 **0 貢獻**（不加）
- \(k\)：**平滑常數**（常見預設 **60**）
  - \(k\) 越大：排名差異被「壓得更平滑」，融合結果更保守
  - \(k\) 越小：更強調很前面的名次差異

最後用 \(\text{RRF}(d)\) **由大到小**排序，得到融合後的候選順序。

---

## 3) 為什麼有人說 RRF「比較穩」？

相對於「把 BM25 raw score 跟 vector distance/similarity 做 min-max normalize 再加權」這類做法，RRF 的穩定感常來自：

- **不吃不同分數量綱**：它只看 rank，不看 raw score
- **比較不受極端值（outlier）影響**：min-max 很容易被某次 query 的分數分佈拉歪

但它仍然是啟發式：**不保證最優**，只是工程上很好落地、好 A/B。

---

## 4) 需要模型呼叫嗎？

**不需要。**RRF 是純算法：只要你拿得到每一路的排序（或每路 top-N 的名次），就能算。

它跟 **rerank**（cross-encoder / LLM rerank）不同：

- **RRF**：融合多路排序（通常仍在「召回後的排序/融合」階段）
- **rerank**：用更強的 query–doc 交互模型去「重打分」（通常更準、但更貴）

---

## 5) 跟你目前 N1 pipeline 的關係（概念對照）

你現在的 hybrid（N1-P1~P3）做的是「分數合併 + normalize + 加權」的一種融合方式。

如果把融合改成 RRF，本質上是把 **融合規則**換掉： (也是融合多路排序（通常仍在「召回後、精排前」或「取代 min-max 加權」）

- 仍然是：keyword 取一批、vector 取一批 → 去重 → **融合排序** →（可選）rerank → finalize

- 你現在等於已經有兩條「夠用且互補」的路：

  - hybrid（N1-P1~P3）：BM25 + vector → 去重 → min-max normalize + 加權排序（偏工程可控、可解釋）
  - rerank / finalize（N1-P4~P5，選用）：在候選更完整後，再用模型把「最後要餵給生成的內容」變得更可靠、更可追溯

  - RRF 就當成未來你想做 「更不吃量綱、對 outlier 更穩」 的融合替代方案備案即可；不急著上也不影響你現在的主線。
